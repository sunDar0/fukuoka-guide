# 가족 여행 검수 (경계면 교차검증)

산출물이 "그럴듯한가"가 아니라 "실제로 맞물리는가"를 검증한다. 핵심은 **경계면 교차 비교**다.

## 왜 경계면인가

버그는 보통 한 파일 안이 아니라 파일 사이에서 난다. 일정 템플릿이 "△△ 신사 방문"이라 적었는데 `data.js`에 그 명소가 없거나, 좌표가 엉뚱하거나, 12월에 닫는 곳이면 — 각 파일은 멀쩡해 보여도 합치면 깨진다. 그래서 한쪽만 보지 않고 **템플릿 ↔ data.js ↔ 좌표 ↔ 영업시간**을 동시에 대조한다.

## 검증 시점 (점진적 QA)

전체 완성 후 1회가 아니라 두 번 돈다.
1. **1차 (설계 대조)**: `01_curator_shortlist.md` + `02_route_plan.md`가 나오면 — 동선이 큐레이션 풀과 맞는지, 일일 이동이 무리 없는지.
2. **2차 (코드 경계면)**: integrator가 `data.js`/`planner.js`를 반영한 뒤 — 템플릿 활동이 실제 명소를 가리키는지, 구문이 깨지지 않았는지.

## 경계면 체크 (스크립트로)

손으로 추정하지 말고 Bash로 파싱해 대조한다.

- **명소명 매칭**: `02_route_plan.md`(또는 `itineraryTemplates`)의 각 활동이 참조하는 명소명이 `data.js`의 `locations[].name`에 실제로 존재하는가. grep으로 대조.
- **좌표 거리**: 인접 활동 명소의 lat/lng로 직선거리를 계산(위경도 차 × 약 111km/도)해 하루 누적 이동이 이동 수단 대비 과하지 않은지. 렌터카라도 하루 장거리 연속은 유아·노부모에 무리.
- **구문 검증**: `node --check data.js && node --check planner.js`로 반영 후 깨짐 확인.
- **필드 정합**: 추가된 명소 객체가 스키마 전 필드를 갖췄는지, region이 8개 허용값 안인지.

## 4대 제약 체크리스트

각 day·활동을 아래로 훑어 통과 못 하는 항목을 표로 잡는다.

| 제약 | 점검 |
|---|---|
| 유아(2살) | childFriendly, 수유실/유아차, 낮잠 슬롯 확보, 좁은 좌석·긴 계단 회피 |
| 노부모 | 앉을 곳·실내·엘리베이터, 하루 도보·서있기 과다 아님, 무리한 조기 출발 아님 |
| 시즌(겨울) | 야외 위주 회피, 휴장·계절한정 확인, 일루미네이션 등 겨울 가점 반영 |
| 이동 수단 | 렌터카면 주차 가능·자동차 소요시간 적정, 짐 옮기는 날 이동 최소화 |

## 산출물 형식 (`_workspace/03_qa_report.md`)

발견 항목별로 심각도·위치·문제·근거·제안을 단다.

```markdown
## 차단 (반드시 수정)
- [Day2 14:00 "○○ 정원"] data.js에 명소 없음 → 동선이 빈 명소 참조. 제안: data.js 추가 또는 활동 교체
- [Day3] 좌표 거리 합계 ~180km → 온천 왕복 + 근교가 하루에 과다. 제안: 근교를 Day2로 분산

## 주의 (검토 권장)
- [Day1 09:00 출발] 노부모·유아에 이른 출발. 제안: 10:30로 조정

## 정보
- planner.js node --check 통과
```

## 하지 않을 것

- 코드를 직접 수정하지 않는다. 무엇이 틀렸는지와 제안만 낸다.
- "명소가 존재한다"에서 멈추지 않는다. 좌표·시간·시즌까지 맞물리는지 본다.
- 상충 정보를 임의로 지우지 않는다. 출처를 병기해 보고한다.
